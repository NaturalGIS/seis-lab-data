{# Flash messages via localStorage. Why not Starlette's session? SSE handlers (e.g. webapp.handlers.handle_project_deletion_success_detail_page) run during the streaming phase of an SSE response. By that point, the response headers have already been sent, and Starlette's SessionMiddleware serialises the session into the Set-Cookie header at that moment — so any session writes that happen later, inside the stream, are silently lost. Why not URL query parameters? The message would be visible in the browser's address bar and would persist in history. The solution: before redirecting, the SSE handler calls execute_script to write the flash payload to localStorage under the key 'sld:flash' (namespaced to avoid collisions with other libraries). localStorage survives full page navigations (unlike DataStar signals, which are cleared on navigation). On the next page load, the script below reads and immediately deletes the value, then dynamically creates and shows a Bootstrap toast. #}
